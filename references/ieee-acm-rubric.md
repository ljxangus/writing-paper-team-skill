# IEEE/ACM 审稿评分标准（Rubric）

本文档定义 Reviewer Agent 使用的 IEEE/ACM 顶会顶刊审稿评分标准。

---

## 评分维度与权重

| 维度 | 权重 | 说明 |
|------|------|------|
| Technical Quality | 30% | 技术正确性、实验严谨性、方法合理性 |
| Originality / Novelty | 25% | 新颖性、与现有工作的本质区别 |
| Significance / Impact | 20% | 问题重要性、对领域的影响 |
| Presentation / Clarity | 15% | 写作质量、结构逻辑、图表质量 |
| Reproducibility | 10% | 实验可复现性、代码/数据可用性 |

---

## 评分等级定义

### Overall Rating

| 等级 | 含义 | 百分位 | 描述 |
|------|------|--------|------|
| Strong Accept | 强烈接受 | top 5% | 论文在该领域做出杰出贡献，几乎没有弱点 |
| Accept | 接受 | top 20% | 论文做出重要贡献，弱点较少且不影响核心结论 |
| Weak Accept | 弱接受 | top 40% | 论文有一定贡献，但存在可改进的弱点 |
| Borderline | 边界 | 50% | 论文有一定价值，但弱点与优点基本持平 |
| Weak Reject | 弱拒绝 | bottom 40% | 论文有明显弱点，需较大改进才能达到接受标准 |
| Reject | 拒绝 | bottom 20% | 论文存在严重问题，无法在当前形式下接受 |
| Strong Reject | 强烈拒绝 | bottom 5% | 论文存在根本性问题 |

### Confidence Level

| 等级 | 含义 | 描述 |
|------|------|------|
| 5: Expert | 专家 | 审稿人对该领域非常熟悉，能评估技术细节 |
| 4: High | 高 | 审稿人对该领域比较熟悉，能评估核心方法 |
| 3: Medium | 中 | 审稿人对该领域有一定了解，能评估整体思路 |
| 2: Low | 低 | 审稿人对该领域了解有限 |
| 1: None | 无 | 审稿人不熟悉该领域 |

---

## 各维度评分细则

### Technical Quality (30%)

| 分数 | 标准 |
|------|------|
| 5 (Excellent) | 方法论严谨，实验设计合理，统计正确，无技术错误 |
| 4 (Good) | 方法论基本合理，实验较为完善，有少量可改进之处 |
| 3 (Fair) | 方法论有一定问题，实验不够完善，但不影响核心结论 |
| 2 (Poor) | 方法论有明显缺陷，实验不够公平或充分 |
| 1 (Very Poor) | 方法论有根本性错误，实验结果不可信 |

**检查要点**：
- 对照组是否充分且公平？
- 评估指标是否全面且合适？
- 统计检验是否正确执行？
- 消融实验是否覆盖关键模块？
- 威胁到有效性是否讨论？

### Originality / Novelty (25%)

| 分数 | 标准 |
|------|------|
| 5 (Excellent) | 提出全新的问题或方法，与现有工作有本质区别 |
| 4 (Good) | 在现有工作基础上有显著创新，非简单组合 |
| 3 (Fair) | 有一定创新性，但与现有工作区别有限 |
| 2 (Poor) | 增量改进，与现有工作区别不明显 |
| 1 (Very Poor) | 无新贡献，基本复现已有工作 |

**检查要点**：
- 核心贡献声明是否明确？
- 与最相关 baseline 的本质区别？
- 技术方案是否是 trivial 的组合？
- 贡献声称是否与实际成果匹配？

### Significance / Impact (20%)

| 分数 | 标准 |
|------|------|
| 5 (Excellent) | 解决领域重要问题，影响广泛，可能引发后续研究 |
| 4 (Good) | 解决有意义的问题，对领域有一定影响 |
| 3 (Fair) | 解决了一定问题，但影响范围有限 |
| 2 (Poor) | 问题不够重要，或解决方案的实用价值有限 |
| 1 (Very Poor) | 问题不重要，或解决方案无实际价值 |

**检查要点**：
- 研究问题是否是领域痛点？
- 方法是否可迁移到其他场景？
- 改进幅度是否有实际意义？
- 是否可能影响该领域的发展方向？

### Presentation / Clarity (15%)

| 分数 | 标准 |
|------|------|
| 5 (Excellent) | 写作清晰流畅，结构严谨，图表优秀，去AI化程度高 |
| 4 (Good) | 写作基本清晰，结构合理，图表可读 |
| 3 (Fair) | 写作尚可但有改进空间，部分图表不够清晰 |
| 2 (Poor) | 写作不够清晰，结构混乱，图表质量差 |
| 1 (Very Poor) | 写作难以理解，结构严重缺失 |

**检查要点**：
- 语言是否流畅？去AI化程度如何？
- 章节组织是否合理？
- 图表是否清晰且有必要？
- 术语是否一致？
- 引用是否规范？

### Reproducibility (10%)

| 分数 | 标准 |
|------|------|
| 5 (Excellent) | 实验描述完整到可直接复现，代码/数据公开 |
| 4 (Good) | 实验描述较为完整，有少量细节需推断 |
| 3 (Fair) | 实验描述基本可复现，但缺少一些关键细节 |
| 2 (Poor) | 实验描述不够详细，难以复现 |
| 1 (Very Poor) | 实验描述严重不足，完全无法复现 |

**检查要点**：
- 超参数是否完整列出？
- 数据集是否有明确来源？
- 代码是否公开或可获取？
- 随机种子是否固定？
- 运行环境是否有记录？

---

## 评分校准指南

### 避免 Leniency Bias

- 不要因为论文写作好就忽略技术问题
- 不要因为领域相关就降低创新性要求
- "没有明显问题" ≠ "应该接受"

### 避免 Harshness Bias

- 不要因为不熟悉子领域就否定论文
- 不要要求超出论文范围的实验
- 小问题不应导致大扣分

### 评分一致性

- Weaknesses 的严重程度应与评分一致
- 3 个 Major Weakness 不应给出 Accept
- 仅有 Minor Issues 不应给出 Reject
- 如评分偏离直觉，应在报告中解释理由
